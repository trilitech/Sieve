IAM

Two layers, one model:

  • Rules are grants (allow / deny). A role is a reusable bundle of rules; a token is issued for one or more roles and can do the union of them. Deny always wins; anything not allowed is denied.
  • Guardrails are global constraints — approval and response filters that always apply to a matching allowed request, no matter which rule (or role) granted it. They can't be bypassed by composing another role.

Filter & guard library

Reusable, named transforms and guards. Author one here, then attach it on a rule (in the rule builder's “Response filters & guards” — it applies whenever that grant is used) or on a guardrail (it applies to every matching request, no matter which rule granted it). Attaching on a rule is safe: obligations from all matching rules are unioned, so composing another role can never strip one.

  • Redact and Exclude items run after the operation — they transform the response the agent receives.
  • Script guard runs before the operation — it inspects the request and can allow, deny, or require approval. This is how you control “what can be sent”.

When several transforms apply to one response they run in ascending order — lower runs first (e.g. excludes 10, redactions 20, scripts 30). Leave blank for the per-kind default.

Runs after the operation, only within the content fields of whichever connector you attach it to (subject, body, …) — never ids, metadata, or base64 attachments. One pattern per line; a field matches if it matches any line. Redact replaces every match with [REDACTED]; Exclude items drops matching items from a list response. Pick Contains for a plain substring or Regex for an RE2 pattern (e.g. \b\d{16}\b for card numbers).

Absolute path to a script file on the Sieve host, under the allowed scripts directory (/opt/sieve-py by default; no ..). It runs before the operation via the sandboxed interpreter, with a 5-second timeout.

Reads a JSON request on stdin:

{"operation": "send_email", "connector": "google",
 "connection": "work",
 "params": {"to": ["a@x.com"], "subject": "...", "body": "..."}}

Writes a JSON decision on stdout: {"action": "allow"}, {"action": "deny", "reason": "..."}, or {"action": "approval_required"}. Anything else — bad output, an exception, or a timeout — is treated as deny (fail-closed).

Example — block sends whose body contains “secret”:

import sys, json
req = json.load(sys.stdin)
body = (req.get("params") or {}).get("body", "")
if "secret" in body.lower():
    print(json.dumps({"action": "deny", "reason": "contains 'secret'"}))
else:
    print(json.dumps({"action": "allow"}))

Absolute path to a script file on the Sieve host, under the allowed scripts directory (/opt/sieve-py by default; no ..). It runs after the operation via the sandboxed interpreter (5-second timeout) and rewrites the response the agent receives.

Reads a JSON request on stdin whose metadata.response holds the response so far (a JSON string):

{"metadata": {"phase": "post",
   "response": "{\"emails\": [{\"id\": \"1\", \"ssn\": \"123-45-6789\"}]}"}}

Writes {"rewrite": "<new response string>"} on stdout — that string replaces the response. (Empty/no rewrite = pass through; an error or timeout fails closed.) Stacks with other transforms in rank order.

Example — structured PII scrub (a real one could call a local LLM):

import sys, json
req = json.load(sys.stdin)
resp = (req.get("metadata") or {}).get("response", "")
data = json.loads(resp)
SENSITIVE = {"ssn", "secret", "token"}
def walk(v):
    if isinstance(v, dict):
        return {k: ("[redacted]" if k in SENSITIVE else walk(x)) for k, x in v.items()}
    if isinstance(v, list): return [walk(x) for x in v]
    return v
print(json.dumps({"rewrite": json.dumps(walk(data))}))
