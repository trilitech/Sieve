The problem

Agents are useful when they can act.

They can triage email, manage infrastructure, update GitHub issues, call APIs, draft replies, inspect logs, manage campaigns, and operate business workflows.

But useful agents need credentials, and raw credentials are a bad boundary.

Gmail scopes are too coarse. API keys are usually all-or-nothing. IAM can be powerful, but it is painful to configure per agent, per project, per task. MCP tools expose capabilities, but they do not automatically give you the policy granularity you need.

And prompt instructions are not permissions.

If an agent is compromised — by prompt injection, a malicious document, a poisoned webpage, a rogue tool, or a leaked conversation — the damage is determined by what the agent can actually access.

Sieve limits that blast radius.

What Sieve does

Sieve sits between agents and services. The agent gets a Sieve token, not the real credential. You decide what that token is allowed to do — which services it can reach, which operations are permitted, and what content the agent can see.

What you get

Composable. Use one feature or all of them.

Keep credentials away from agents

Agents talk to Sieve. Sieve talks to your services. Your real passwords, keys, and tokens never leave Sieve's hands.

Define fine-grained policies

Create rules narrower than the underlying service usually allows. Match on labels, content, models, regions, recipients, costs.

Filter, redact, and script

Block operations up front, or inspect responses and redact sensitive content before the agent sees it. When rules aren't enough, write a short Python script for any custom logic.

Add human approval where it matters

Hold sensitive operations — sending email, changing infra, posting to production, calling expensive APIs — in an approval queue.

Rotate once, not everywhere

Give every agent its own Sieve token. When access changes, update the role or revoke the token. Your service credentials stay untouched.

Audit every agent action

Sieve logs which token made each request, which connection it used, what operation it attempted, what policy decision was made, and how long it took.

Define exactly what each agent can do

Email assistant

Help with email, but don't own the inbox.

The agent can find relevant messages and draft replies. It cannot send without approval, and it cannot see sensitive content.

role: email-assistant
policies:
  - service: gmail
    action: read_email
    allow:
      labels:
        - project-x

  - service: gmail
    action: create_draft
    allow: true

  - service: gmail
    action: send_email
    require_approval: true

  - service: gmail
    action: read_email
    deny_if:
      content_contains:
        - CONFIDENTIAL
        - salary
        - medical
AWS staging operator

Routine staging ops, never production.

The agent can debug staging. It cannot touch production, and it cannot create dangerous network rules.

role: staging-operator
policies:
  - service: aws
    allow:
      actions:
        - cloudwatch.read_logs
        - ecs.restart_service
        - s3.read_object
      environment: staging

  - service: aws
    deny:
      environment: production

  - service: aws
    deny_if:
      security_group_allows: "0.0.0.0/0"

When rules aren't enough, write a script.

Write a short Python script. It sees the full request and response, and returns allow, deny, redact, or escalate. Any rule the service's own permissions can't express, yours can.

Filter responses

Redact PII before the agent sees it.

Strip SSNs, credit cards, and phone numbers from every email body. The agent gets the content it needs for triage; the sensitive patterns are replaced with [REDACTED] before they ever reach the model.

redact_pii.py script policy 
#!/usr/bin/env python3
import json, sys, re

req = json.load(sys.stdin)
phase = req.get("metadata", {}).get("phase", "pre")

if phase != "post":
    print(json.dumps({"action": "allow"}))
    sys.exit(0)

response = req["metadata"].get("response", "")

patterns = [
    (r"\d{3}-\d{2}-\d{4}",       "[SSN REDACTED]"),
    (r"\d{4}[-\s]?\d{4}[-\s]?\d{4}[-\s]?\d{4}", "[CC REDACTED]"),
    (r"\+?1?[-.\s]?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}", "[PHONE REDACTED]"),
]

redacted = response
for pattern, replacement in patterns:
    redacted = re.sub(pattern, replacement, redacted)

print(json.dumps({"action": "allow", "rewrite": redacted}))
Scope visibility

Only show emails from approved domains.

An agent helping with client work shouldn't see your personal mail, HR threads, or recruiter spam. This script drops any email not from a domain you've approved.

domain_allowlist.py script policy 
#!/usr/bin/env python3
import json, sys

ALLOWED = {"@acme-client.com", "@company.com"}

req = json.load(sys.stdin)
phase = req.get("metadata", {}).get("phase", "pre")

if phase != "post":
    print(json.dumps({"action": "allow"}))
    sys.exit(0)

data = json.loads(req["metadata"].get("response", "{}"))

if "emails" in data:
    data["emails"] = [
        e for e in data["emails"]
        if any(e.get("from", "").endswith(d) for d in ALLOWED)
    ]
    data["total"] = len(data["emails"])

print(json.dumps({"action": "allow", "rewrite": json.dumps(data)}))
LLM-powered policy

Ask a local model if an email is sensitive.

Some rules are hard to express as patterns. This script sends the email to a local Ollama model and asks whether it contains sensitive information. If the model says yes, the email is blocked before the agent ever sees it.

sensitivity_check.py script policy 
#!/usr/bin/env python3
import json, sys
from ollama import chat

req = json.load(sys.stdin)
phase = req.get("metadata", {}).get("phase", "pre")

if phase != "post":
    print(json.dumps({"action": "allow"}))
    sys.exit(0)

email = json.loads(req["metadata"].get("response", "{}"))
body = email.get("body", "")

resp = chat(model="llama3", messages=[{
    "role": "user",
    "content": f"Does this email contain sensitive information "
               f"(financials, legal, HR, medical, credentials)? "
               f"Answer YES or NO only.\n\n{body[:2000]}"
}])

answer = resp.message.content.strip().upper()

if answer.startswith("YES"):
    print(json.dumps({
        "action": "deny",
        "reason": "Email flagged as sensitive by local model"
    }))
else:
    print(json.dumps({"action": "allow"}))
Built in

Agents request their own permissions.

Instead of you configuring every rule by hand, the agent can propose the policy it needs. You review and approve it — or reject it. Faster than writing rules from scratch, and the agent gets exactly the access it asked for, nothing more.

The agent calls propose_policy with the rules it wants. The proposal lands in the approval queue. You read it, decide if it makes sense, and click approve. Done.

agent's MCP tool call 
{
  "name": "propose_policy",
  "arguments": {
    "name": "email-triage-readonly",
    "description": "I need to read emails labeled
      support-queue to triage tickets. No send,
      no drafts, no access to other labels.",
    "default_action": "deny",
    "rules": [
      {
        "match": {
          "operations": ["list_emails", "read_email"]
        },
        "action": "allow"
      }
    ]
  }
}
This goes to your approval queue. The agent can't enact it — you decide.
